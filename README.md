# AI Coding From Idea To Product

这是《从想法到产品：普通人的 AI 编程实战》的配套开源项目仓库。

目标很明确：让没有专业编程背景的读者，也能跟着文章看懂一次需求怎么被 AI 辅助开发出来，再打开项目、验证结果，并把同样的过程迁移到自己的产品想法里。

## 学习方式

这个仓库不是让读者先跑一串测试。

新读者先打开 `product-playground`，把自己的产品想法填进去，看到一条从第一个页面、官网、Web App、H5、插件、轻量后端到上线检查的跟练路线：

```bash
cd product-playground
npm start
```

浏览器打开：

```text
http://127.0.0.1:5172
```

先看见产品路线，再进入具体模块。`npm test`、`npm run verify`、dry-run 和 smoke test 是后面的证据层，用来证明页面、接口、权限、模型或上线边界没有被改坏，不是学习体验的第一步。

每个关键产品项目默认包含三类交付：


```text
页面 / 弹窗 / 清单   # 读者先看到的产品结果
process/   # 需求开发全过程记录
src/       # 最终可运行代码或核心实现
```

`process/` 不是为了增加文档负担。产品实战看的是需求开发过程，不是执行一个 `js` / `ts` 命令看输出。示例项目会尽量保留完整过程，方便你复盘一个需求怎样从想法、边界、上下文、AI 任务、小步实现走到验证和 Review。你自己做产品时，先写清 `idea`、`spec`、`plan` 和 `verification` 就够开始第一轮；任务变复杂后，再补上下文、提示词、实现记录和 Review。

建议跟练顺序：

1. 先运行 `product-playground`，把自己的想法映射到产品路线。
2. 进入对应模块，先打开页面、弹窗或清单，走一条用户路径。
3. 再看 `process/01-idea.md` 和 `process/02-spec.md`，理解为什么这样切第一版。
4. 继续看 `process/05-plan.md`、`process/06-implementation-log.md` 和 `process/08-review.md`，理解 AI 任务怎么拆、哪里不能夸大。
5. 最后运行 `npm run verify`，确认测试、页面、dry-run、权限或清单证据都还成立。

页面结果负责让读者愿意继续学，过程记录负责训练 AI 编程判断力，验证命令负责守住边界。

## 主线模块

| 模块 | 作用 |

| `product-playground/` | 把读者自己的想法转成可视化跟练路线 |
| `tools-lab/` | Codex App / CLI 工作台自检 |

基础概念、工作流、总览、复盘、附录这类文章不单独造 npm 项目；它们围绕第一个产品和实战项目讲。真正需要运行和验证的内容放在对应产品模块里。运行命令只是验证手段，不是实战主体；产品实战模块要优先保留需求开发过程记录。

## 学完后怎么开自己的项目

不要从复制某个示例源码开始。先新建自己的产品目录，然后准备这几样东西：

```text
README.md
.env.example
AGENTS.md
process/01-idea.md
process/02-spec.md
process/05-plan.md
process/07-verification.md
```

第一轮只做一条主路径，并把过程记录下来：

```text
写清目标用户和问题
切出第一版 MVP
做出第一个可操作界面
跑一次页面、接口、命令或测试验证
记录下一轮要改什么
```

这几步跑通以后，再考虑真实模型、数据库、登录、支付、部署和更复杂的自动化。

## 和一键生成工具的关系

Lovable、Bolt、Replit、秒哒这类工具很适合快速看到第一版页面。你可以用它们找感觉。

这个仓库解决的是下一步：第一版出来以后，怎么持续改、怎么知道 AI 改了哪些文件、怎么验证页面没有坏、怎么保护 API Key、怎么判断 demo 离可试用产品还差什么。也就是说：

```text
一键生成工具让你更快看到第一版
这套项目训练你把第一版继续做下去
```

所以这里不会追求“最短时间生成最炫页面”。它更关心你能不能把一个想法稳定推进成自己的产品。

## 专栏主线

```text
模块 0 导读：为什么普通人现在可以开始做产品
模块 1 先跑起来：工作台、国内可用性判断、第一个网页工具、运行验证、Git 回退
模块 2 补够用基础：终端、Node/npm、API、.env、工具入口
模块 3 用 AI 工具做产品：MVP、spec、项目规则、上下文、任务切片、验证
模块 4 产品实战：网站、网页应用、H5/小程序、插件、轻量后端
模块 5 上线和迁移：部署、SEO、成本、支付沙箱、反馈和迁移
附录：工具模型表、错误速查、提示模板和产品 idea
```

## 不是什么

这个仓库不是专业程序员深入工程专栏，也不是所有 AI 工具和工程框架的大合集。

这个仓库只保留一个原则：能帮助读者上手开发自己的产品，就进入这里；只适合专业工程实现的内容，后续单独承接，不在这里展开。

因此这里不会把复杂框架和自动化工具作为入门必装项。它们可以在你熟悉流程之后作为进阶工具使用；在这个仓库里，默认先用页面、弹窗、清单、README、`process/`、提示模板和验证证据把主路径跑稳。

## 基础知识怎么学

这个项目不会把基础知识写成传统教材。

每个基础概念都跟着项目出现：

- 跑第一个网页工具时，理解目录、文件、浏览器、localhost 和 npm 脚本。
- 第一次让 AI 改代码时，理解 Git diff、保存版本和回退。
- 接入模型时，理解 API Key、`.env`、HTTP 请求、错误码、费用和兼容边界。
- 做网页、H5、小程序、插件和 App 时，理解页面、状态、数据、权限和发布路径。
- 上线前，理解日志、权限、部署、回滚、SEO、支付和成本。

读者不需要先学完全部编程基础再开始。更合理的方式是：遇到项目任务，补够用基础，马上运行和验证。

## 环境变量

复制 `.env.example` 为 `.env`，按需填写自己的模型服务配置。

```bash
cp .env.example .env
```

真实 API Key 不要写进前端代码，不要提交到 Git，也不要出现在截图里。

## 验证原则

每个主线模块必须提供：

- 一个读者能先看到的结果：页面、弹窗、清单、dry-run 报告或判断矩阵。
- 启动命令和 `npm run verify`。
- README。
- `.env.example`，如果涉及配置。
- 过程记录。
- 常见错误说明。
- 对应文章编号或主题。

如果某个模块还未实现，README 必须明确写成“规划中”，不能伪装成已经可运行。
