# 05 实现计划

## 计划

1. 先写核心生成规则的测试。
2. 实现 `buildBrief`，让测试通过。
3. 创建页面结构。
4. 接上表单提交和结果渲染。
5. 增加复制提示词按钮。
6. 写 README。
7. 运行测试。
8. 打开浏览器做真实交互验证。

## 为什么先写测试

页面可以靠浏览器看，但生成规则不应该每次都靠手点。

`buildBrief` 是这个项目最重要的规则：它决定了从模糊想法到产品简报的转换方式。先用测试保护这条规则，后面让 AI 修改字段、默认值或提示词时，就不会完全靠肉眼判断。

## 为什么不先做样式

第一版最重要的是链路成立：

```text
输入 -> 生成 -> 展示 -> 复制 -> 验证
```

样式只要清楚、可读、适合截图即可。过早打磨视觉，会让第一个项目偏离“AI 编程过程”这个主线。

## 风险点

- 表单字段名和 `buildBrief` 入参不一致，导致生成内容为空。
- 复制按钮在没有生成结果时可点击，用户体验不清楚。
- 直接把用户输入塞进 HTML，留下 XSS 风险。
- README 里的命令和真实脚本不一致。

## 对应处理

- 表单字段统一使用 `idea`、`audience`、`coreFeatures`、`style`、`constraints`。
- 复制按钮默认禁用，生成结果后再启用。
- 渲染前对字符串做 HTML 转义。
- 完成后实际运行 `npm test` 和浏览器验证。
