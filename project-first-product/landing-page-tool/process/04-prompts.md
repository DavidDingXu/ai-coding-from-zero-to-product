# 04 关键提示词

## 第一次生成项目骨架

```text
请在当前目录创建一个零依赖的本地网页工具，产品名叫“AI 产品简报生成器”。

目标：
把用户输入的模糊产品想法整理成结构化产品简报，并生成一段可以交给 AI 编程工具继续开发的提示词。

约束：
1. 使用原生 HTML、CSS、JavaScript。
2. 不引入前端框架和第三方依赖。
3. 核心逻辑放在 src/brief-builder.js，页面交互放在 src/app.js。
4. 使用 Node 内置 node:test 写测试。
5. README 里写清楚 npm test、npm start 和浏览器访问地址。

请先给出实现计划，再按计划创建文件。完成后运行测试，并说明如何用浏览器验证。
```

## 为什么这样写

这段提示词没有直接说“帮我做个网页”。它补了五个边界：

- 目标：生成产品简报和 AI 编程工具提示词。
- 技术约束：原生 Web，不引入依赖。
- 文件边界：核心逻辑和页面交互分开。
- 验证方式：必须有测试。
- 完成方式：先计划，再实现，再验证。

AI 编程里，提示词不是许愿。提示词的作用是把任务边界讲清楚。

## 让 AI 修改小功能的提示词

```text
请阅读当前项目。我要给产品简报增加一个“上线前检查”字段。

要求：
1. 表单里增加一个输入框，提示用户填写上线前需要检查的事项。
2. buildBrief 输出的 brief 里包含“上线前检查”。
3. 如果用户不填，默认使用“先确认页面可打开、核心功能可点击、测试通过”。
4. 先补测试，再改实现。
5. 改完后运行 npm test，并说明如何在浏览器里验证。
```

## 不推荐的提示词

```text
帮我优化一下这个工具。
```

这个说法太宽。AI 可能改样式、换结构、加依赖、改函数名，也可能把已经能跑的东西改坏。

更好的做法是小步修改，每一步都说明输入、输出、默认值和验证方式。
