# 06 implementation log

## 第 0 轮：先收住需求

原始想法是做「AI 个人效率助手」官网。这里先没有让 AI 直接生成完整页面，而是先把本轮范围压到 5 件事：

```text
产品一句话说明
核心使用场景
三步工作流
候补表单
本地表单校验
```

同时明确排除登录、真实模型、远程数据库、支付和自动提醒。

这个判断很重要：官网阶段只验证陌生用户能不能看懂产品，并愿意留下最小反馈。它还不能证明今日计划生成质量，也不能证明用户愿意注册或付费。

## 第 1 轮：测试先行

新增：

```text
test/landing-page.test.js
package.json
```

运行：

```bash
npm test
```

红灯结果：

```text
ERR_MODULE_NOT_FOUND
Cannot find module .../src/landing-page.js
```

红灯符合预期，说明测试已经指向还没有实现的官网核心逻辑。

这一步没有先写页面，是因为页面容易把注意力带到布局和视觉上。先用测试锁住范围，可以防止 AI 顺手把官网扩成完整应用。

## 第 2 轮：实现核心逻辑和页面

新增：

```text
src/landing-page.js
src/cli.js
src/app.js
index.html
styles.css
.env.example
README.md
process/
```

核心设计：

- `landing-page.js` 负责文案、表单校验、候补记录和检查清单。
- `app.js` 只负责把核心逻辑接到浏览器页面。
- `cli.js` 让 README 的命令可以直接验证。
- 页面只做本地结果展示，不上传远程服务。

实现时特意把 `storage: local-demo` 写进候补记录，防止文章或页面把本地 demo 说成远程保存。

## 第 3 轮：检查 AI 容易跑偏的点

本轮重点检查了 3 个风险：

- 是否偷偷引入真实请求或远程保存。
- 是否要求用户填写 API Key、密码或 token。
- 是否把后续“生成今日计划”的能力写成当前已经实现。

最终保留的做法是：官网只承诺后续会进入今日计划应用；当前表单只收集角色、场景和痛点，并拦截密钥类文本。

## 第 4 轮：验证和复盘

验证分三层：

- 测试证明文案、校验、隐私边界和候补记录结构存在。
- CLI 输出证明 brief、feedback 和 checklist 与 README 对齐。
- 浏览器页面证明首屏、空表单、错误输入和完整表单能被用户实际操作。

下一轮进入 `web-app`，才开始开发今日计划生成、筛选、复制和本地保存。
