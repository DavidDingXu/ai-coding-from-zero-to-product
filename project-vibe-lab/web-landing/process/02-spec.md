# 02 spec

## 目标

完成「AI 个人效率助手」官网第一版，读者能本地运行页面并完成一次候补表单验证。

这一轮要验证的真实需求是：陌生用户第一次看到这个产品时，能不能理解它解决什么问题，并留下一个足够支撑下一轮判断的反馈。

不是验证：

- 模型是否能生成今日计划。
- 用户是否愿意注册账号。
- 数据是否能云端保存。
- 支付或订阅是否成立。

## 范围

本轮要做：

- 产品首屏。
- 三个价值说明。
- 三步流程。
- 候补表单。
- 本地输入校验。
- 页面内成功 / 失败提示。
- CLI 输出 brief、copy、feedback、checklist。
- 测试覆盖核心文案、表单校验、隐私边界和 AI 提示词。

本轮不做：

- 登录和账号体系。
- 真实模型调用。
- 远程数据库保存。
- 支付和会员系统。
- 自动发送真实提醒。

## 用户和场景

目标用户：

- 学生：任务、课程、资料分散，不知道今天先做什么。
- 上班族：会议、待办、聊天记录分散，容易遗漏关键动作。
- 产品经理和运营：每天信息源多，需要快速整理今日重点。
- 独立开发者：同时处理开发、运营、内容和账目，希望有一个轻量计划入口。

首屏必须回答：

```text
这是什么产品
它帮谁解决什么问题
第一版现在能承诺什么
哪些能力当前还没有
```

## 验收标准

- 首屏一句话不夸大，不写成已经有真实模型自动处理全部生活事务。
- 表单只收集称呼、邮箱、角色、关注场景和当前痛点。
- 表单拦截空字段、错误邮箱和密钥类文本。
- 成功记录必须标明 `storage: local-demo`。
- 页面、README、测试和文章对“本地 demo、不上传远程服务”的说法一致。

## 验证方式

```bash
npm test
npm run brief
npm run feedback
npm run checklist
npm start
```

浏览器访问：

```text
http://127.0.0.1:5174
```

页面检查：

- 空表单会提示缺少字段。
- 错误邮箱会被拦截。
- 含 `sk-`、`api_key`、`password`、`secret`、`token` 的输入会被拦截。
- 完整表单会生成本地候补记录。
