# 06 实现记录

## 第 0 轮：先做移动端取舍

`web-app` 的完整流程是输入上下文、生成 10 条候选、筛选、复制和保存。手机端不能照搬这条路径。

本轮先把需求压成：

```text
短输入 -> 3 条推荐 -> 本地收藏 -> 行动文本
```

这一步的判断是：移动端用户更可能在碎片时间完成一个动作，而不是管理完整计划库。

## 红灯

先写 `test/h5-miniapp.test.js`。

第一次运行 `npm test`，失败原因是核心模块不存在：

```text
Cannot find module '../src/h5-miniapp.js'
```

补一个空模块后，再次运行测试，失败落到业务断言：项目边界、输入归一化、推荐生成、行动计划、收藏、分享文本、发布检查和提示词都还未实现。

这一步确认测试不是摆设，它确实在约束移动端产品边界。

## 绿灯

实现 `src/h5-miniapp.js` 后，`npm test` 通过。

随后补充：

```text
src/cli.js
src/app.js
index.html
styles.css
.env.example
```

页面和 CLI 复用同一个核心模块，避免浏览器逻辑和命令行验证各写一套规则。

## 取舍

本轮没有接真实模型。

原因是这一步训练的是移动端产品压缩和发布路径判断。模型接入会把读者注意力拉到 API Key、费用、网络和错误码上，反而冲淡本节目标。

本轮也没有做真实小程序上传审核。

原因是小程序涉及主体、类目、隐私协议、域名备案、体验版测试和平台审核。当前项目只把这些作为发布路径检查项，不把 H5 demo 写成已上线小程序。

## 检查容易跑偏的点

- 是否把 Web App 的 10 条候选和筛选器照搬到手机端。
- 是否把 H5 本地页面写成已经提审通过的小程序。
- 是否引入 openid、云数据库或订阅消息。
- 是否让用户在前端填写真实模型 Key。

最终保留的是本地 H5 主路径，验证产品压缩，不验证平台发布能力。
