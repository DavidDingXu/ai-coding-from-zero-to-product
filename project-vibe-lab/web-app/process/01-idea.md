# 01 原始想法

`web-landing` 已经证明「AI 个人效率助手」的官网入口能打开，也能用候补表单收集反馈。下一步不能停在“产品看起来存在”，而是要让用户真的输入上下文，得到今日行动卡，并做一次取舍。

这个需求来自一个真实使用场景：用户每天的信息分散在任务清单、日程、聊天记录、笔记和零散账目里。真正需要的不是另一个空白 ToDo，而是把这些信息收拢后，给出今天可以先执行的几张行动卡。

本项目只解决一个问题：

```text
输入任务、日程、笔记、账目和今日目标后，生成 10 张今日行动卡，并支持筛选、复制和本地保存。
```

第一版不用真实模型。原因很简单：如果本地规则都不能把页面、筛选、保存和验证跑通，直接接模型只会把问题藏到 API 后面。

目标用户仍然是：

- 效率工具用户
- 独立开发者
- 运营人员
- 想用 AI 做自己产品的新手

这轮项目要让读者看到：AI 编程不是从“接模型”开始，而是从“把流程做成可验证产品”开始。

真实约束：

- 当前不能要求用户填写 API Key。
- 当前不能读取真实日历、账单、聊天记录或通讯录。
- 当前保存只做浏览器本地记录，不承诺多端同步。
- 当前生成用确定性规则模拟，不把它写成真实模型能力。

下一轮反馈问题：

- 用户是否愿意输入任务、日程、笔记和账目。
- 行动卡字段是否足够帮助用户取舍。
- 用户更想保存单张行动卡，还是整份今日计划。
- 哪些输入必须在接模型前补上校验。
